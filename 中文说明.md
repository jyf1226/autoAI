# AGENTS.md 中文说明

本文档是对 `AGENTS.md` 的中文化解读，方便在项目里快速理解 oh-my-codex (OMX) 的执行规则与协作方式。

## 1. 这份文件是做什么的

- `AGENTS.md` 是当前工作区的顶层行为契约（operating contract）。
- 任何角色提示（如 `prompts/*.md`）都要服从它，不能覆盖它。
- 核心目标：在保证安全与可验证性的前提下，提高多代理协作效率与交付质量。

## 2. 核心工作原则

- 能直接完成就直接完成，避免无意义委派。
- 只有在“明显提升质量/速度/正确性”时才委派给子代理。
- 沟通要短、具体、可执行。
- 先证据后结论：声明完成前必须验证。
- 默认优先轻路径：直接操作 > MCP > 子代理。
- 对不熟悉的 SDK/框架/API，先查官方文档再实现。

## 3. 工作协议（Working agreements）

- 做清理/重构前，先写清理计划。
- 行为可能受影响时，先补回归测试再改代码。
- 倾向“删除”而不是“新增”。
- 优先复用已有工具与模式，避免新抽象泛滥。
- 未经明确要求，不新增依赖。
- 保持改动小、可审阅、可回滚。
- 改完后执行 lint/typecheck/tests/static analysis。
- 最终汇报要包含：改了哪些文件、做了哪些简化、剩余风险是什么。

## 4. Lore 提交规范（Commit message）

提交信息强调“为什么改”，不是“改了什么”。

建议结构：

1. 第一行：意图（why）
2. 正文：背景与决策依据
3. 可选 trailers，例如：
  - `Constraint:` 外部约束
  - `Rejected:` 被否方案及原因
  - `Confidence:` 信心等级
  - `Scope-risk:` 影响范围风险
  - `Directive:` 给后续维护者的提醒
  - `Tested:` 已验证内容
  - `Not-tested:` 尚未验证缺口

## 5. 模式与委派规则

默认是“直接执行”。只有需要时才切模式：

- `$deep-interview`：需求边界不清、不能假设时先澄清，不做实现。
- `$ralplan`：需求已大体清楚，但方案/权衡/测试策略需要共识。
- `$team`：多并行任务且需要持续协调时使用团队编排。
- `$ralph`：单 owner 持续推进直到完成并验证。

子代理规则要点：

- 并发子代理最多 6 个。
- 子代理必须受 `AGENTS.md` 约束。
- worker 不应私自重写全局计划或擅自切模式。
- 发生共享文件冲突、权限不足、范围扩大等情况要上报 leader。

## 6. 关键词触发与技能路由

用户消息命中关键词时会自动映射到对应技能（不再二次确认）。

常见映射示例：

- `autopilot` -> `$autopilot`
- `ralph` -> `$ralph`
- `plan this` -> `$plan`
- `deep interview` / `don't assume` -> `$deep-interview`
- `code review` -> `$code-review`
- `security review` -> `$security-review`
- `cancel` / `stop` -> `$cancel`

注意：`autopilot`、`ralph`、`ultrawork`、`team/swarm` 等属于 **OMX runtime 工作流**。如果当前不在 OMX runtime 环境，需要提示并切换到可用替代路径，或按用户要求从 shell 启动 OMX。

## 7. 模型路由（Model routing）

按任务复杂度选择角色与模型能力：

- 低复杂度：`explore`、`writer` 等快速角色
- 中复杂度：`executor`、`debugger`、`test-engineer`
- 高复杂度：`architect`、`critic`、`executor`

团队模式下模型有优先级解析（显式参数 > 继承 leader > 默认 spark/frontier 配置），避免随意猜测默认模型。

## 8. 验证与收尾要求

完成声明前必须走验证闭环：

1. 明确“什么证据能证明完成”
2. 实际执行验证
3. 读取输出并基于证据汇报
4. 若失败则继续迭代，不提前收工

规模建议：

- 小改动：轻量验证
- 常规改动：标准验证
- 大改动/安全架构类：深度验证

结束前应确认：

- 无未完成项
- 功能正常
- 测试通过
- 无已知错误
- 验证证据齐全

## 9. 状态与目录约定（.omx）

运行态数据主要在 `.omx/`：

- `.omx/state/`：模式状态
- `.omx/notepad.md`：会话笔记
- `.omx/project-memory.json`：跨会话记忆
- `.omx/plans/`：计划文档
- `.omx/logs/`：日志

生命周期要求：

- 启动时写入 state
- 阶段变化时更新 state
- 完成时标记 inactive 并记录完成时间
- 取消/中止时清理状态

## 10. 快速上手

- 执行 `omx setup` 安装组件
- 执行 `omx doctor` 做健康检查
- 在任务中按需使用 `$deep-interview`、`$ralplan`、`$team`、`$ralph` 等工作流
